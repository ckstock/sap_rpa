SAP 自动化门户本地开发与部署实施方案

在没有服务器的阶段，先完成前端门户、本地 C# Bridge、NCo 报表数据、SAP GUI 自动化的本机闭环；后续有服务器后，前端可静态部署，执行能力迁移到 Windows Server 或 Windows Worker。

01

结论摘要

推荐路线

当前最合理实施方式

你现在正在走的是“静态网页/Netlify + 用户本机 SapWebLauncher 协议唤醒”的现行方案。这个方案适合先发布前端、先让本机 SAP GUI Script 跑起来，但它只是触发链路，不是完整任务闭环。下一步应在 SapWebLauncher 外面补一层本地 Bridge API，让网页能读取任务状态、日志、导出文件和报表结果。

当前缺口：拿不到脚本执行后的报表结果状态

单纯点击网页链接触发 sap-zck:// 或 sap-automation:// 后，浏览器无法知道 VBS 是否启动成功、SAP 是否登录成功、事务是否保存成功、报表是否导出成功。要在网页上显示“执行中/成功/失败/报表数据”，必须增加 http://127.0.0.1 本地 API、WebSocket/SSE，或未来服务器回调。

核心边界

能力负责模块说明
页面展示与任务提交Web 前端负责配置、触发、状态展示，不保存 SAP 密码。
RFC/BAPI/报表数据NCo 数据层NCo 是 RFC 层登录，不等于 SAP GUI 登录。
事务码前台操作SAP GUI Script用于必须点界面、保存、导出、处理弹窗的事务。
任务状态与结果回传本地 Bridge API单纯协议唤醒没有返回值，网页看结果必须有 API 或后续服务器接口。

平台约束

NCo 建议放 Windows 本机、Windows Server 或 Windows Worker。若未来必须 Linux 后端承载 RFC 层，需要评估 JCo；但 SAP GUI Script 仍必须落在 Windows + SAP GUI 环境。

03

ALV 远程 DUMP 处理

准确结论

不是绝对不能做，而是要看源码路径

如果程序唯一出口就是创建前台 CL_GUI_ALV_GRID / custom container，远程 RFC/NCo 没有 GUI 容器，确实会 DUMP，不能靠 NCo 参数硬修。但如果源码存在后台、list、export、SALV 分支，或者能在 ALV 显示前拿到内表，就可能通过控制入参或源码级取数实现。

可能实现

可能路径源码特征实现方式结论
隐藏参数切换输出模式p_list、p_batch、no_display用 selection-table 切到 list/background/export 分支。可行性高
后台 Job 触发非 GUI 分支判断 sy-batch = 'X' 后走 spool/list。提交后台作业，读取 job/spool 结果。可行性高
SALV/list runtime 抓取使用 CL_SALV_TABLE、REUSE_ALV 或 list 输出。包装函数里 set runtime info，再 SUBMIT 并 get data ref。需 POC
显示前内表导出调用 ALV 前已形成 gt_out、gt_alv、lt_data。封装 remote-enabled FM，直接返回结果内表。最稳定
动态读全局内表结果表是报表全局变量。动态 ASSIGN 报表全局表后序列化。谨慎使用

排查方法

源码搜索清单

打开目标 Z 报表源码，搜索 CALL SCREEN、CREATE OBJECT ... CONTAINER、SET_TABLE_FOR_FIRST_DISPLAY、REUSE_ALV、CL_SALV、WRITE、sy-batch、EXPORT TO MEMORY。先确认 DUMP 点和结果内表位置，再决定能否通过入参绕过 GUI。

优先级方案适用场景
P0ABAP 抽取数据出口核心报表、需要稳定上线、网页要展示结构化数据。
P1报表增加后台模式ABAP 能改原程序，且原逻辑可复用。
P1后台 Job + Spool报表支持后台执行，输出格式稳定。
P2cl_salv_bs_runtime_info 抓取SALV/list 输出、无 GUI container 强依赖。
P3Windows GUI Worker 导出没有 ABAP 配合，短期必须跑通。

总体架构

方案全景对比

方案做法优点限制结果状态回传能力
方案 A0：静态网站 + SapWebLauncher
现行方案		前端可以部署到 Netlify；用户电脑安装并注册 SapWebLauncher，网页通过协议链接唤醒本机 SAP GUI Script。最轻量、无需服务器、现在已有基础，适合先演示和小范围试运行。协议调用是单向触发；终端 SAP GUI、注册表、权限、脚本环境差异要逐台治理。无闭环
网页只能发起动作，不能可靠知道脚本结果和报表内容。
方案 A1：静态网站 + 本机 Bridge API
推荐下一步		保留 Netlify/静态前端和 SapWebLauncher，新增监听 127.0.0.1 的本地 API，由 Bridge 统一管理任务、日志、截图、导出文件和 NCo 数据。仍然不需要服务器，同时网页可以展示执行状态、失败原因、报表结果。要开发本地 API，并处理 CORS、Private Network Access、请求签名和端口占用。可闭环
前端轮询 /api/tasks/{taskId} 或订阅日志流。
方案 B：Windows Server 集中执行把 API、NCo、SAP GUI Worker 放到一台 Windows Server 或专用执行机，前端只调用服务器。任务、日志、权限、版本和执行环境集中管理。需要服务器、SAP GUI 会话治理、并发隔离和运维能力。可闭环
服务器数据库/API 持久化状态和结果。
方案 C：Linux 后端 + Windows WorkerLinux 负责门户、普通 API、调度和数据库；Windows Worker 负责 NCo 与 SAP GUI Script。适合公司已有 Linux 后端体系，普通业务服务更好部署。NCo 与 GUI Script 不应直接规划为 Linux 执行组件，仍要维护 Windows Worker。可闭环
Linux 负责状态中心，Windows Worker 回传任务结果。
方案 D：纯 SAP 接口/NCo/ABAP 出口ABAP 封装 RFC/BAPI/自定义 FM，直接返回内表或 JSON；前端通过后端读取结构化数据。最稳定、最适合报表数据展示，不依赖前台 GUI。需要 ABAP 配合；不能覆盖必须前台保存、弹窗确认、GUI-only 的事务步骤。数据闭环强
报表结果天然可结构化返回。

对你当前方案的判断

Netlify 可以发布静态网页，且不会影响本机 SapWebLauncher 的协议唤醒；但 Netlify 不能替你执行 SAP、不能读取本机 VBS 日志、不能拿到 SAP 报表数据。现行方案要补齐“脚本执行后的报表结果状态”，最小改造不是上服务器，而是把 SapWebLauncher 升级或包装成 127.0.0.1 本地 Bridge。

系统分层

Web 门户

Vue/Vite 静态站点，负责流程展示、任务提交、状态查看

↓
本地 SAP Bridge

C# Windows 进程，提供本地 API、协议唤醒、任务状态和日志

↓
NCo 数据层

调用 RFC/BAPI/自定义 FM，获取结构化报表数据

↓
SAP GUI 执行层

sapshcut + VBS/GUI Script，处理必须前台执行的事务

↓
结果回传

本地 API / WebSocket / 未来服务器 API 返回任务状态与报表数据

本地 API 必要性

协议唤醒不等于数据回传

sap-zck:// 或 sap-automation:// 只能让网页发起本机执行，天然没有返回值。你需要获取 SAP 报表数据并展示在网站里，因此本地 Bridge 必须提供 http://127.0.0.1 API 或 WebSocket。

GET http://127.0.0.1:17890/health POST http://127.0.0.1:17890/api/tasks GET http://127.0.0.1:17890/api/tasks/{taskId} GET http://127.0.0.1:17890/api/tasks/{taskId}/logs GET http://127.0.0.1:17890/api/tasks/{taskId}/result

最小升级路径

  1. 前端继续按静态站做，先发布到 Netlify 或本地 IIS。
  2. SapWebLauncher 保留协议唤醒能力，但每次执行生成 taskId。
  3. 新增本地 Bridge API，把 VBS 启动、SAP 登录、事务执行、截图、导出文件、NCo 报表查询都写入任务状态。
  4. 网页启动任务后轮询 http://127.0.0.1:17890/api/tasks/{taskId}，拿到 running/success/failed 和报表结果。
  5. 后续有服务器时，把任务状态中心迁移到 Windows Server 或 Linux + Windows Worker，不需要推翻前端。

安全要求

Bridge 只监听 127.0.0.1，不要监听 0.0.0.0。CORS 只允许本地开发地址和未来正式域名。SAP 密码不写进网页、URL、localStorage 或日志。

08

部署与风险

部署路线

路线做法优点缺点适用判断
路线 A0：Netlify 静态网站 + 本机 SapWebLauncher
现行方案		前端部署到 Netlify；每台执行电脑安装 SapWebLauncher 并注册协议。无服务器也能推广，最接近当前实现。不能回传脚本执行状态和报表结果。适合演示、验证点击启动 SAP 的链路。
路线 A1：静态网站 + 用户本机 Bridge API
推荐落地		前端仍静态部署；每台执行电脑安装 Bridge，监听 127.0.0.1。无服务器，同时能拿状态、日志、截图、导出文件和报表结果。终端环境差异要治理，本地 API 安全要做。适合你当前没有服务器但要做完整闭环。
路线 B：Windows Server 集中执行API、NCo、GUI Worker 放 Windows Server/执行机。日志、权限、任务集中。需要服务器和运维。适合多人使用、审计要求高、统一执行环境。
路线 C：Linux 后端 + Windows WorkerLinux 承载普通 API/调度，Windows Worker 执行 SAP GUI。适应公司 Linux 架构。NCo 与 GUI Script 仍要落 Windows Worker。适合未来公司要求标准后端部署。
路线 D：纯 SAP 接口/NCo/ABAP 出口ABAP 返回结构化报表数据，GUI Script 只保留必要事务。数据稳定、性能好、最利于网页展示。需要 ABAP 开发配合。适合关键报表和高频查询长期治理。

风险控制

风险影响控制办法
SAP 密码暴露URL、localStorage、日志泄露。前端不传密码；Bridge 用凭据管理、加密配置或 SSO。
ALV 远程 DUMPNCo 无法后台抓取部分报表。逐报表分型；必要时 ABAP 出口或 GUI 导出。
本地 API 被滥用恶意网页可能调用 Bridge。只监听 127.0.0.1；CORS allowlist；请求签名；action 白名单。
SAP GUI 不稳定弹窗、超时、控件变更导致失败。步骤日志、截图、重试、控件路径版本管理。
平台限制NCo 不能直接规划为 Linux 组件。NCo 放 Windows；Linux 只做普通后端时保留 Windows Worker。

推荐演进顺序

先保留路线 A0 作为已跑通的触发入口；马上推进路线 A1，补本地 Bridge API，解决报表结果状态回传；当任务量、权限、日志审计要求上来后，再演进到路线 B。路线 C 只有在公司强制 Linux 后端时再考虑，路线 D 用来治理核心报表数据。
